# 如何理解 Q 的形状：$h_q \times s_q \times d_k$

> 这篇文档回答一个问题：**为什么 Q 的形状是 $h_q \times s_q \times d_k$（head 数 × query 数 × head 维）？这三个维度分别代表什么？为什么要这样组织？**
>
> 面向零基础读者。知道矩阵乘就行，不需要懂注意力机制。

---

## 一、问题背景：Q 是什么？为什么关心它的形状？

### 1.1 Q 在大模型里扮演什么角色？

Transformer 模型里有一个核心操作叫**注意力（Attention）**，简单说就是：

> 模型在生成每个词时，要"回看"历史上下文，决定哪些历史词更重要、哪些不重要。

这个"回看"的过程分三个角色：

| 张量 | 全名 | 比喻 | 实际作用 |
|------|------|------|---------|
| **Q**（Query） | 查询 | "我现在想问的问题" | 当前要生成的 token，带着"我要找什么"的提问 |
| **K**（Key） | 键 | "档案柜里每个文件的标签" | 历史每个 token 的"特征索引"，用来匹配 Q |
| **V**（Value） | 值 | "档案柜里每个文件的内容" | 历史每个 token 的"实际信息"，匹配度高的会被取出 |

注意力计算的本质：

```
1. 用 Q 去和所有 K 比对 → 得到"相似度分数"
2. 用分数对 V 做加权平均 → 得到最终输出
```

**Q 的形状决定了"这次要问几个问题、问得多详细"**——这就是为什么我们要搞清楚它的形状。

### 1.2 为什么是三个维度？

一个数（标量）只有一个维度。一串数（向量）有两个维度（比如 1×128）。一个矩阵有两个维度（比如 64×128）。

**Q 为什么要有三个维度？** 因为它要同时表达三件不同的事情：

1. 有几个"提问者"在同时问 → head 数 $h_q$
2. 每个提问者一次问几个问题 → query 数 $s_q$
3. 每个问题问得多细致 → head 维 $d_k$

下面分别展开。

---

## 二、三个维度分别是什么？

### 2.1 第一维 $h_q$：head 数（"提问者"数量）

**什么是 head？**

Transformer 用"多头注意力"（Multi-Head Attention）。一个 head 就是一个**独立的注意力计算单元**——它有自己的 Q、K、V，独立算相似度和加权。

**为什么要有多个 head？**

想象你看一篇文章：
- 一个 head 关注"语法"——主谓宾结构对不对
- 另一个 head 关注"语义"——意思连不连贯
- 再一个 head 关注"指代"——"他"指的是谁

**每个 head 学到不同的关注模式，多个 head 协作得到更全面的理解**。

| 模型 | $h_q$（Q 的 head 数） |
|------|---------------------|
| BERT-base | 12 |
| GPT-3 (175B) | 96 |
| LLaMA-2 (70B) | 64 |
| **DeepSeek V3（解码）** | **128** |

DeepSeek V3 解码用 128 个 head，意味着 128 个"提问者"在同时关注上下文的不同方面。

**MLA 的特殊之处**：传统注意力里 K 和 V 也要有相同的 head 数（每对 Q-K head 一一配对）。MLA 把 K 和 V 压缩成一个共享的 latent 表示，所以 K 的 head 数 $h_k$ 可以远小于 $h_q$。但 Q 仍然有 128 个 head——每个 head 都要独立算一次注意力。

### 2.2 第二维 $s_q$：query 数（一次算几个 token）

**$s_q$ 是什么？**

每个 head 里，这次要并行算几个 query token。

**两种典型场景**：

| 场景 | $s_q$ 的值 | 解释 |
|------|-----------|------|
| **普通解码** | 1 | 每次只生成 1 个新 token，所以只有 1 个 query |
| **预填充（Prefill）** | 几百到几千 | 一次性处理整个 prompt，所有 prompt token 都是 query |
| **推测解码（Speculative）** | 几个到几十个 | 一次猜多个 token，再批量验证 |

**DeepSeek V3 解码时 $s_q = 1$**：每生成一个字，Q 只有 1 个 token——"我现在要生成第 N+1 个字，请看前 N 个字帮我决定"。

**为什么 $s_q$ 重要？**

$s_q$ 直接决定了计算量。每个 query token 都要和**所有** KV token 算相似度，所以：

```
总计算量 ∝ s_q × s_k（query 数 × KV 数）
```

$s_q = 1$（解码）vs $s_q = 4096$（预填充），计算量差 4096 倍——这就是为什么解码和预填充是两种完全不同的性能场景。

### 2.3 第三维 $d_k$：head 维（每个问题的"细致度"）

**$d_k$ 是什么？**

每个 query token 在每个 head 里，用一个 $d_k$ 维的向量表示。$d_k$ 就是这个向量的维度。

**为什么要有 $d_k$ 维？**

一个 token 不能只用一个数表示——太粗糙。要用一个高维向量才能表达丰富的语义。

| 模型 | $d_k$（head 维） |
|------|----------------|
| BERT-base | 64 |
| GPT-3 | 128 |
| **DeepSeek V3（MLA）** | **512** |

**$d_k$ 在数学上的作用**：

Q 和 K 的相似度用**点积**算：

$$
\text{score} = Q \cdot K = \sum_{i=1}^{d_k} Q_i \cdot K_i
$$

$d_k$ 越大，点积能比较的维度越多，相似度判断越精细。但 $d_k$ 越大计算量也越大（点积要做 $d_k$ 次乘加）。

**MLA 的 $d_k = 512$ 是怎么回事？**

传统 MHA 里 $d_k$ 比较小（64~128），因为 K 和 V 要在每个 head 里各存一份，$d_k$ 大了显存爆炸。MLA 把 K 和 V 共享成一个 latent 表示，显存大幅节省，所以可以用更大的 $d_k = 512$——每个 head 能表达更丰富的信息。

## 三、三个维度组合起来：一个三维张量

把三个维度放一起：

```
Q 的形状：[h_q, s_q, d_k]

  h_q 个 head（外层）
    └─ 每个 head 里有 s_q 个 query token（中层）
        └─ 每个 query token 是一个 d_k 维向量（内层）
```

### 3.1 用一个具体例子感受

DeepSeek V3 解码配置：$h_q = 128, s_q = 1, d_k = 512$。

```
Q 是一个三维张量，形状 [128, 1, 512]
  
  128 个 head
    每个 head 里 1 个 query token
      每个 token 是 512 维向量

总元素数 = 128 × 1 × 512 = 65,536 个数
BF16 下占内存 = 65,536 × 2 = 131,072 字节 ≈ 128 KB
```

### 3.2 三层"嵌套"结构

可以这样想象 Q：

```
Q[h_q=128][s_q=1][d_k=512]

  ┌──── h_q = 128 个"提问者" ────┐
  │                              │
  │  head 0:                     │
  │    ┌─ s_q = 1 个问题 ─┐      │
  │    │  [v0, v1, ...v511] │    │  ← d_k = 512 维向量
  │    └──────────────────┘      │
  │                              │
  │  head 1:                     │
  │    ┌─────────────────┐       │
  │    │  [v0, v1, ...v511] │    │
  │    └─────────────────┘       │
  │                              │
  │  ...                         │
  │                              │
  │  head 127:                   │
  │    ┌─────────────────┐       │
  │    │  [v0, v1, ...v511] │    │
  │    └─────────────────┘       │
  │                              │
  └──────────────────────────────┘
```

**每个 head 独立做注意力**，所以 Q 的第一维 $h_q$ 是"批处理"维度——所有 head 并行计算，互不干扰。

## 四、Q 的形状在代码里长什么样？

### 4.1 FlashMLA 接口里的实际定义

看 FlashMLA 的 Python 接口（`flash_mla/flash_mla_interface.py`）：

```python
q: (batch_size, seq_len_q, num_heads_q, head_dim)
```

**等等，这里有 4 个维度，公式里只有 3 个？**

差的那个维度是 **batch_size**——一次处理多个序列的批处理维度。在 FlashMLA 的算-访比推导里，我们关心的是**单个序列**的计算量，所以省略 batch 维。完整形状是：

| 维度 | 公式符号 | 实际含义 |
|------|---------|---------|
| 0 | （省略） | batch_size：一次处理几个序列 |
| 1 | $s_q$ | seq_len_q：每个序列的 query token 数 |
| 2 | $h_q$ | num_heads_q：每个序列的 head 数 |
| 3 | $d_k$ | head_dim：每个 head 的向量维度 |

**所以完整 Q 形状 = `(batch, s_q, h_q, d_k)` = `(B, s_q, h_q, d_k)`**

公式里省略 batch 后是 $h_q \times s_q \times d_k$（顺序也调整了，head 在最外层方便推导）。

### 4.2 接口代码里的真实注释

FlashMLA 接口源码（`flash_mla_interface.py`）里这样写：

```python
sched_meta.config = FlashMLASchedMeta.Config(
    q.shape[0],       # batch_size：这次同时处理几个序列？
    q.shape[1],       # seq_len_q：每个序列的 Q 长度（解码时通常是 1）
    q.shape[2],       # num_heads_q：Q 有几个"头"（MLA 是 128 或 64）
    k_cache.shape[1], # page_block_size：一个 KV 缓存块存几个 token
)
```

**这里清楚地展示了 4 个维度的对应关系**——每一维都有实际的工程意义。

## 五、为什么是这三个维度？少一个行不行？

### 5.1 假设没有 head 维（只剩 $s_q \times d_k$）

```
Q 形状: [s_q, d_k]   ← 只有一个"提问者"
```

后果：模型只能学到一种关注模式（比如只关注语法）。表达能力大幅下降。

**多头注意力的核心价值就是用多个 head 学多种模式**。少了 head 维，模型就退化成"单头注意力"，效果差很多。

### 5.2 假设没有 seq 维（只剩 $h_q \times d_k$）

```
Q 形状: [h_q, d_k]   ← 每次只算 1 个 token
```

后果：预填充阶段一次性算几千个 token 时，必须循环几千次。**性能灾难**。

把 $s_q$ 作为独立维度，让 GPU 可以并行处理多个 query token——这是高效推理的基础。

### 5.3 假设没有 head_dim 维（只剩 $h_q \times s_q$）

```
Q 形状: [h_q, s_q]   ← 每个 token 用一个标量表示
```

后果：token 之间只能用"一个数"区分，无法表达丰富语义。模型直接失效。

**$d_k$ 维是 token 语义表达的载体**，少不得。

**结论：三个维度各有不可替代的作用，缺一不可。**

## 六、Q 形状如何决定计算量？

回顾一下 MLA 解码的算-访比公式（详见 [FlashMLA-MLA计算访存比推导详解.md](FlashMLA-MLA计算访存比推导详解.md)）：

$$
\text{FLOPs} = 2 h_q s_q s_k (d_k + d_v)
$$

把这个公式和 Q 的形状 $h_q \times s_q \times d_k$ 对照看：

| 公式中的因子 | 来自哪里 | 含义 |
|------------|---------|------|
| $h_q$ | Q 的第 1 维 | 每个 head 都要算一次 |
| $s_q$ | Q 的第 2 维 | 每个 query token 都要算一次 |
| $d_k$ | Q 的第 3 维 | 点积要算 $d_k$ 次 |
| $s_k$ | 来自 K | KV Cache 里的每个 token 都要比对 |
| $d_v$ | 来自 V | 输出向量的维度 |

**Q 的三个维度 $h_q, s_q, d_k$ 都直接进了 FLOPs 公式**——每一维都是计算量的乘法因子。这就是为什么搞清楚 Q 的形状这么重要：**它直接决定了 kernel 要做多少次浮点运算**。

### 6.1 一个直觉口诀

> **Q 的形状 = "多少个提问者 × 一次问几个问题 × 每个问题多细致"**
>
> 三个维度都是计算量的乘法因子，每一维翻倍，计算量就翻倍。

---

## 七、Q 的形状在不同场景下怎么变？

同样的模型，不同场景下 Q 的形状差异巨大：

### 7.1 对比表

| 场景 | $h_q$ | $s_q$ | $d_k$ | Q 总元素数 | 计算量量级 |
|------|-------|-------|-------|-----------|----------|
| DeepSeek V3 解码（普通） | 128 | 1 | 512 | 65,536 | 小 |
| DeepSeek V3 解码（开 MTP，猜 4 个 token） | 128 | 4 | 512 | 262,144 | 中 |
| DeepSeek V3 预填充（4096 token prompt） | 128 | 4096 | 512 | 268M | 巨大 |
| 小模型解码（LLaMA-7B，TP=8） | 4 | 1 | 128 | 512 | 极小 |

**几个观察**：

1. **$h_q$ 和 $d_k$ 由模型架构决定**，不会因场景改变
2. **$s_q$ 是场景变量**——解码小、预填充大、推测解码中等
3. **预填充的 $s_q$ 是解码的几千倍**，所以预填充是计算密集型，解码（普通情况下）是访存密集型——但 MLA 因为 $h_q = 128$ 很大，解码也变成计算密集型

### 7.2 为什么 $h_q$ 不会变？

$h_q$ 是模型定义时就定好的（DeepSeek V3 是 128）。**Tensor Parallel 会切分 $h_q$**：

- TP=1（DeepSeek 解码默认）：单卡看到全部 128 个 head
- TP=8：每张卡只看到 16 个 head

**这就是为什么 DeepSeek V3 解码不用 TP**——保持 $h_q = 128$ 才能让计算-访存比超过 roof-line 门槛，变成计算密集型（详见 [FlashMLA-计算访存比详解.md](FlashMLA-计算访存比详解.md)）。

## 八、用图形直观感受 Q 的形状

### 8.1 解码场景（$s_q = 1$）

```
Q 形状: [128, 1, 512]

head 0   head 1   head 2   ...   head 127
  ↓        ↓        ↓                ↓
[v]      [v]      [v]              [v]    ← 每个 head 只有 1 个 512 维向量

总共 128 个 512 维向量
```

### 8.2 预填充场景（$s_q = 4096$）

```
Q 形状: [128, 4096, 512]

head 0            head 1           ...   head 127
  ↓                 ↓                       ↓
[v][v][v]...[v]   [v][v][v]...[v]         [v][v][v]...[v]
← 4096 个 →      ← 4096 个 →             ← 4096 个 →

总共 128 × 4096 = 524,288 个 512 维向量
```

### 8.3 一个 token 在 Q 里到底是什么？

假设我们解码生成第 100 个字"猫"，"猫"对应的 embedding 经过模型投影后：

- **在 head 0 里**是一个 512 维向量 $[v_0, v_1, \ldots, v_{511}]$
- **在 head 1 里**是另一个 512 维向量（不同的投影）
- ...
- **在 head 127 里**是又一个 512 维向量

**同一个 token 在不同 head 里有不同的向量表示**——因为每个 head 学到的关注模式不同，所以投影方式也不同。

## 九、常见误区与 FAQ

### Q1：$h_q$（head 数）和 $d_k$（head 维）什么关系？是不是 $h_q \times d_k$ = 隐藏维度？

**是的，对传统 MHA 成立**。比如 BERT-base：$h_q = 12, d_k = 64$，$h_q \times d_k = 768$ = 隐藏维度。

**但 MLA 不严格成立**：MLA 的 K/V 共享 latent 表示，$h_q \times d_k$ 可以大于隐藏维度。DeepSeek V3 的 $h_q = 128, d_k = 512$，乘积 65,536 远大于隐藏维度——这是 MLA 通过共享 latent 实现的"参数效率"。

### Q2：为什么 Q 的形状公式里 $d_k$ 用 K 的下标？Q 的维度不应该叫 $d_q$ 吗？

**好问题**。在标准注意力里，Q 和 K 必须可以点积，所以 Q 和 K 的向量维度必须相同——习惯上统一记作 $d_k$（"key 维度"）。

**MLA 中**：Q 的维度严格说是 $d_q$，K 的维度是 $d_k$，但因为 Q×K^T 要求 $d_q = d_k$，所以推导里都记作 $d_k$。

### Q3：$s_q$ 和 batch_size 不是一回事吗？

**不是**。容易混淆，区分一下：

| 维度 | 含义 | 比喻 |
|------|------|------|
| batch_size | 一次处理几个**独立的序列** | 同时服务几个客户 |
| $s_q$ | 每个序列里**这次要算几个 token** | 给一个客户一次处理几件商品 |

举例：batch_size=4, s_q=1 表示"同时给 4 个客户各生成 1 个字"。batch_size=1, s_q=4096 表示"给 1 个客户一次性处理 4096 个字"（预填充）。

### Q4：Q 在内存里是行优先还是列优先？维度顺序重要吗？

**重要**。FlashMLA 接口里 Q 的形状是 `(batch, seq_len_q, num_heads, head_dim)`，**head 维在 token 维之后**。这个顺序是为了：

1. 同一个 token 的不同 head 在内存里连续（方便 TMA 一次搬一个 token 的所有 head）
2. 不同 token 的同一个 head 不连续（但注意力计算时一个 head 一个 head 算，这个访问模式可接受）

**公式推导时为了简洁，写成 $h_q \times s_q \times d_k$（head 在外层）**，但实际内存布局是 `(batch, s_q, h_q, d_k)`（head 在中间）。这两个表达描述的是同一个东西，只是维度顺序不同。

### Q5：Q 的形状跟 KV Cache 大小有什么关系？

**没有直接关系**。KV Cache 的大小由 $s_k$（KV token 数）和 $d_k$（head 维）决定，和 Q 的形状无关。

但 Q 的形状决定了**计算量**：$h_q \times s_q$ 越大，对同一份 KV Cache 要做的计算越多。

### Q6：解码时 $s_q = 1$，那 Q 不是很小吗？为什么 FlashMLA 还说计算密集型？

**关键在 $h_q = 128$**。虽然 $s_q = 1$，但 128 个 head 都要算一遍，计算量被放大 128 倍。而 KV Cache 虽然大但被 128 个 head 共享——搬运量不变，计算量 ×128，所以算-访比顶过了 roof-line 门槛。

详见 [FlashMLA-MLA计算访存比推导详解.md](FlashMLA-MLA计算访存比推导详解.md)。

### Q7：Q 的 dtype 会影响形状吗？

**不影响形状，但影响内存占用**。

| dtype | 每元素字节数 | DeepSeek V3 解码 Q 的内存 |
|-------|------------|-------------------------|
| FP32 | 4 | 256 KB |
| BF16 | 2 | 128 KB |
| FP8 | 1 | 64 KB |

FlashMLA 主要用 BF16 和 FP8。FP8 能把 Q 的内存减半，但形状不变。

---

## 十、一句话总结

> **Q 的形状 $h_q \times s_q \times d_k$ 表达了三件独立的事情：**
>
> - **$h_q$（head 数）**：有多少个"提问者"在并行关注不同方面（DeepSeek V3 = 128）
> - **$s_q$（query 数）**：每个 head 这次要算几个 token（解码 = 1，预填充 = 几千）
> - **$d_k$（head 维）**：每个 token 用多长的向量表示（DeepSeek V3 = 512）
>
> **三个维度都是计算量的乘法因子**，所以 Q 的形状直接决定了 kernel 的 FLOPs。
>
> 完整接口形状是 `(batch, s_q, h_q, d_k)`，公式推导省略 batch 维、调整顺序后写成 $h_q \times s_q \times d_k$——同一个东西的不同视角。

| V（Value） | 注意力中的"值"张量，表示历史 token 的内容 |

| $h_q$ | Q 的 head 数（query head 数量） |
| $s_q$ | 每个 head 的 query token 数 |
| $d_k$ | 每个 head 的向量维度（key 维度） |
| batch_size | 一次处理的独立序列数 |
| MHA | Multi-Head Attention，标准多头注意力 |
| MLA | Multi-head Latent Attention，DeepSeek 提出的高效注意力 |
| TP | Tensor Parallel，张量并行，把 head 切分到多卡 |
