# 计算访存比（FLOPs/byte）—— 小白也能懂

> 这篇文档回答一个问题：**为什么 GPU 内核要分"计算密集型"和"访存密集型"？怎么判断？**
>
> 以 FlashMLA 的 roof-line 分析为例，从头讲清楚"计算-访存比"这个概念。

---

## 一、先讲一个生活比喻：做饭的两种瓶颈

想象你在一家餐厅做饭。一道菜从原料到上桌，需要两件事：

1. **从仓库搬原料到厨房**（搬数据）
2. **在灶台上切、炒、煮**（做计算）

这两种事情都可能成为瓶颈：

| 瓶颈在哪 | 现象 | 你该做什么 |
|---------|------|-----------|
| **搬原料太慢**（仓库远、走道窄） | 灶台经常空着等原料 → **访存密集型** | 加宽走道、提前预约原料 |
| **切炒太慢**（厨子刀工慢） | 原料堆在案板上厨子来不及处理 → **计算密集型** | 升级切菜机、加厨师 |

GPU 内核也是一样：

- **仓库 = 显存（HBM）**：存放模型参数、KV Cache
- **走道 = 显存带宽（Bandwidth）**：每秒能搬多少字节（GB/s）
- **灶台 = 计算单元（Tensor Core / CUDA Core）**：每秒能做多少浮点运算（FLOPS）
- **厨子 = GPU 算力**

**关键问题**：这道菜到底是"搬原料慢"还是"切炒慢"？判断对了，才知道优化哪边。

## 二、计算-访存比是什么？

### 2.1 直觉定义

> **计算-访存比 = 这段代码要做多少次浮点运算 ÷ 这段代码要搬多少字节数据**

单位：**FLOPs/byte**（每搬 1 字节，要做多少次浮点运算）。

读作"算术密度"（Arithmetic Intensity）也行，意思一样。

### 2.2 三个例子先建立直觉

**例 1：把一个向量加 1**

```python
y[i] = x[i] + 1   # 1 次加法
```

- 计算：1 个加法 = 1 FLOP
- 访存：读 1 个 x（4 字节）+ 写 1 个 y（4 字节）= 8 字节
- 比 = 1/8 = **0.125 FLOPs/byte**

→ 几乎全在搬数据，**典型访存密集型**。

**例 2：两个向量点乘**

```python
s += x[i] * y[i]   # 1 次乘 + 1 次加
```

- 计算：2 FLOPs
- 访存：读 x（4 字节）+ 读 y（4 字节）= 8 字节
- 比 = 2/8 = **0.25 FLOPs/byte**

→ 还是访存密集，但比上一个好一点。

**例 3：矩阵乘 `C = A @ B`，A 是 M×K，B 是 K×N**

- 计算：≈ 2MKN FLOPs
- 访存：读 A（MK）+ 读 B（KN）+ 写 C（MN）≈ MK + KN + MN 字节（每个元素 4 字节）
- 当 M=N=K=4096 时：
  - 计算 ≈ 1.3 × 10¹¹ FLOPs
  - 访存 ≈ 2 × 10⁸ 字节
  - 比 ≈ **650 FLOPs/byte**

→ 计算量远大于访存量，**典型计算密集型**。这就是为什么矩阵乘是 GPU 的好朋友。

### 2.3 一个直觉口诀

> **算得多、搬得少 → 计算密集型**
> **算得少、搬得多 → 访存密集型**

## 三、为什么 GPU 上这个比值特别重要？—— Roof-line 模型

### 3.1 GPU 有两个"天花板"

任何 GPU 都有两个性能上限：

| 天花板 | 含义 | H800 的数字 |
|-------|------|------------|
| **算力天花板** | 每秒最多做多少浮点运算 | 865 TFlops = 865 × 10¹² FLOPs/s（降频后） |
| **带宽天花板** | 每秒最多搬多少字节 | 3.35 TB/s = 3.35 × 10¹² 字节/s |

### 3.2 平衡点：让两个天花板恰好相等

GPU 的"平衡点"= 算力 ÷ 带宽：

```
865 TFlops ÷ 3.35 TB/s = 258 FLOPs/byte
```

**这个数字的意思是**：

- 如果一段代码的**计算-访存比 = 258**，那么 GPU 的算力和带宽**同时用满**，理想情况。
- 如果**比值 > 258**，算力先到顶，带宽还有富余 → **计算密集型**（再加带宽也没用）。
- 如果**比值 < 258**，带宽先到顶，算力还有富余 → **访存密集型**（再加算力也没用）。

### 3.3 一张图理解 Roof-line

```
性能上限 (FLOPs/s)
   ^
   |          ____________  ← 算力天花板 865T
   |         /
   |        /
   |       /  ← 这段斜线是带宽天花板
   |      /     (性能 = 比值 × 3.35T)
   |     /
   |    /
   |   /
   +---+------------------> 计算-访存比 (FLOPs/byte)
       258
       ↑
       平衡点
```

- 内核的比值在 **258 左边** → 撞上斜线（带宽限制）
- 内核的比值在 **258 右边** → 撞上水平线（算力限制）

**这个图的形状像房子的屋顶，所以叫 roof-line 模型。**

## 四、FlashMLA 的实例：MLA 解码为什么是计算密集型？

### 4.1 MLA 解码的计算量与访存量

DeepSeek 在博客里给出的推导（简化版）：

| 量 | 公式 | 解释 |
|----|------|------|
| **FLOPs（计算量）** | $2 h_q s_q s_k (d_k + d_v)$ | Q×K 和 score×V 两个矩阵乘 |
| **访存量（bytes）** | $\approx 2 s_k d_k$ | 主要在搬 KV Cache（$s_k$ 远大于其他维度） |
| **计算-访存比** | $2 h_q s_q$ | 把上面两个相除得到的简化结果 |

参数含义：
- $h_q$ = query head 数量
- $s_q$ = 每个请求的 query token 数（解码时 = 1）
- $s_k$ = KV Cache 的 token 数
- $d_k, d_v$ = K/V 的 head 维度

### 4.2 代入 DeepSeek V3 的解码配置

DeepSeek V3 解码的设置：

- $h_q = 128$（不用 Tensor Parallel，128 个 head 全在一张卡上）
- $s_q = 1$（关掉 MTP/推测解码）
- 所以 $h_q s_q = 128$

**计算-访存比 = $2 \times 128 = 256$ FLOPs/byte**

### 4.3 和 GPU 平衡点比较

```
MLA 解码的比值:           256 FLOPs/byte
H800 的平衡点:            258 FLOPs/byte
含 1/2 softmax 因子的门槛: 129 FLOPs/byte
```

**等等，这里有个细节**：

- H800 的"理论平衡点"是 258 FLOPs/byte
- 但注意力计算不是纯矩阵乘，中间还夹着 softmax（指数、求和、归一化等非矩阵运算）
- 经验上把这些非矩阵运算的损失估算为 1/2 因子
- 所以**实际门槛** = 258 ÷ 2 ≈ **129 FLOPs/byte**

MLA 解码的比值 **256** > 门槛 **129** → **计算密集型** ✓

### 4.4 这结论反直觉在哪？

**普通认知**：大模型解码阶段是访存密集型。

理由：解码时 Q 只有 1 个 token，KV Cache 巨大（几千到几万个 token），主要的活儿是"把 KV Cache 从显存搬到计算单元"，应该被带宽卡住。

**MLA 的反直觉**：

- $h_q = 128$ 个 head **同时在算**
- 每个 head 都要做 Q×K 和 score×V
- 计算量被放大了 128 倍，而访存量没变（KV Cache 是共享的）
- 于是计算-访存比被顶上去了

**一句话总结**：MLA 把 128 个 head 的计算量摞在一起，"算得多、搬得少"，所以是计算密集型。

## 五、判断流程：你自己怎么算？

如果你拿到一个新的 kernel，想判断它是计算密集型还是访存密集型，按这 5 步走：

### Step 1：算 FLOPs

数清楚要做多少次乘法和加法。经验公式：
- 矩阵乘 `C = A @ B`，A 是 M×K，B 是 K×N → **2MKN FLOPs**
- 一个 element-wise 操作（加、乘、激活）→ **1 FLOP per element**

### Step 2：算访存量（bytes）

数清楚要从显存读多少字节、写多少字节：
- 读输入张量：`元素数 × 每元素字节数`
- 写输出张量：`元素数 × 每元素字节数`
- BF16 = 2 字节，FP16 = 2 字节，FP32 = 4 字节，FP8 = 1 字节
- 如果某个张量被多次复用，**只算一次读取**（在共享内存或 L2 缓存里命中）

### Step 3：算比值

```
计算-访存比 = FLOPs ÷ bytes
```

### Step 4：查 GPU 平衡点

| GPU | 算力 | 带宽 | 平衡点（FLOPs/byte） |
|-----|------|------|---------------------|
| H100 SXM | 989 TFlops（降频 ~890） | 3.35 TB/s | ~266 |
| H800 SXM | 990 TFlops（降频 ~865） | 3.35 TB/s | ~258 |
| A100 80G | 312 TFlops | 2.0 TB/s | ~156 |
| RTX 4090 | 82.6 TFlops | 1.01 TB/s | ~82 |

**注意**：算力要用"实际可持续峰值"，不是"理论峰值"。H800 因为降频，理论 990 实际只有 ~865。

### Step 5：比较并得出结论

```
if 计算访存比 >= GPU平衡点:
    → 计算密集型，优化算力利用率（让 Tensor Core 满载）
else:
    → 访存密集型，优化访存（少搬数据、提高带宽利用率）
```

**经验因子**：如果 kernel 中间夹着非矩阵运算（softmax、layernorm、激活等），算力会损失，门槛可下浮 1/2 ~ 1/3。

## 六、常见误区与 FAQ

### Q1：FLOPs 和 FLOPS 是一回事吗？

**不是**，差一个字母意思完全不同：

| 写法 | 含义 | 单位 |
|------|------|------|
| **FLOPs**（小写 s） | 浮点运算**次数**（总量） | 次（无量纲） |
| **FLOPS**（大写 S） | 每秒浮点运算**次数**（速度） | 次/秒 |

计算-访存比的分子是 **FLOPs**（总次数），分母是 bytes（总字节数）。

### Q2：计算-访存比高就一定快吗？

**不一定**。比值高只说明"瓶颈在算力"，但如果你算力利用得很差（比如 Tensor Core 大量空闲），实际性能还是低。

FlashMLA 的优化目标就是：既然 MLA 解码是计算密集型，那就让 **Tensor Core 利用率达到 80%+**。最终他们做到了 660 TFlops（≈ 865 的 76%）。

### Q3：带宽和延迟是一回事吗？

**不是**，这是新手最容易混淆的：

| 概念 | 含义 | 类比 |
|------|------|------|
| **带宽（Bandwidth）** | 单位时间能搬多少字节 | 走道有多宽 |
| **延迟（Latency）** | 一次请求从发出到数据就绪的时间 | 走一趟要多久 |

**compute-bound 场景下带宽不是瓶颈，但延迟仍可能成问题**——数据虽然"搬得动"，但"搬得不够及时"，计算单元还是要等。FlashMLA 用**细粒度 TMA pipelining**（把数据切成小块提前发起请求）来隐藏延迟。

### Q4：为什么矩阵乘的计算-访存比这么高？

因为**数据复用**。矩阵乘 `C = A @ B` 中，A 的每个元素要参与 N 次计算，B 的每个元素要参与 M 次计算。同一个数据被多次"用"，所以算得多、搬得少。

相比之下，element-wise 操作（`y[i] = x[i] + 1`）每个数据只被用 1 次，算得少、搬得多。

### Q5：KV Cache 不是很大吗，为什么 MLA 解码还是计算密集型？

因为 **head 数多**。每个 head 都要做 Q×K 和 score×V，128 个 head 把计算量放大了 128 倍。而 KV Cache 虽然大，但它是 128 个 head **共享**的——搬运量没有放大。

**计算量 ×128，访存量 ×1**，所以比值被顶上去。

### Q6：为什么有 1/2 的 softmax 因子？

注意力计算的标准流程是：

```
1. Q × K → score     (矩阵乘，Tensor Core 加速)
2. softmax(score)    (指数、求和、归一化，CUDA Core 干)
3. score × V → O     (矩阵乘，Tensor Core 加速)
```

步骤 2 不是矩阵乘，Tensor Core 帮不上忙，只能走 CUDA Core。这部分"白占"了时间，所以算力利用率打折。经验上打折一半，于是门槛从 258 降到 129。

这也是为什么 FlashAttention 系列拼命优化 softmax 的调度——**让 CUDA Core 做 softmax 的同时，Tensor Core 别闲着**。

## 七、回到 FlashMLA：这个判断决定了什么？

判断 MLA 解码是**计算密集型**之后，所有优化方向就明确了：

| 如果是计算密集型（实际） | 如果是访存密集型（假设） |
|----------------------|----------------------|
| 让 Tensor Core 满载 | 提高 KV Cache 搬运效率 |
| 重叠 CUDA Core 和 Tensor Core | 减少不必要的数据搬运 |
| Seesaw 调度、warpgroup 交错 | 增大 L2 缓存命中率 |
| 目标：算力利用率 80%+ | 目标：带宽利用率 90%+ |

**所以 FlashMLA 的核心创新——seesaw 调度、TMA pipelining、cache hints——全部都是为了"让 Tensor Core 别闲着"。如果一开始判断错了（以为是访存密集型），所有优化方向都会南辕北辙。**

这就是为什么 roof-line 分析是写 GPU kernel 的第一步。

## 八、一句话总结

> **计算-访存比 = 这段代码要做多少次浮点运算 ÷ 要搬多少字节数据**
>
> 比值高于 GPU 平衡点 → 计算密集型 → 优化算力
> 比值低于 GPU 平衡点 → 访存密集型 → 优化访存
>
> MLA 解码的比值 256 FLOPs/byte > H800 门槛 129 FLOPs/byte → 计算密集型
> → 这决定了 FlashMLA 的所有优化都围绕"让 Tensor Core 满载"展开。
